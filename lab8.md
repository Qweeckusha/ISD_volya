## Раздел 1 — Выбор архитектурного стиля

### Выбранный стиль: Клиент–сервер (3-уровневая архитектура)

```
┌─────────────────┐
│   Клиенты       │
│  ┌───────────┐  │
│  │ Веб-браузер│  │ ← сотрудники представительства
│  │ (ПК/планшет)│ │
│  └───────────┘  │
└────────┬────────┘
         │ HTTPS / REST API
         ▼
┌─────────────────┐
│   Сервер приложений │
│  ┌───────────┐  │
│  │ REST API  │  │ ← бизнес-логика, валидация, отчёты
│  │ (бэкенд)  │  │
│  └───────────┘  │
└────────┬────────┘
         │ SQL
         ▼
┌─────────────────┐
│   База данных   │
│  ┌───────────┐  │
│  │   MySQL   │  │ ← хранение всех сущностей из Лаб 7
│  └───────────┘  │
└─────────────────┘
```

### Обоснование выбора

**Почему клиент–сервер подходит:**

1. **Соответствие требованиям из Лаб 6**: 
   - `NFR-USE-02` требует адаптивного дизайна для работы на планшетах и ноутбуках — веб-интерфейс + API легко масштабируется под разные устройства
   - `NFR-COM-01` требует поддержки нескольких браузеров — тонкий клиент (браузер) решает эту задачу
   - `NFR-SEC-01/02` (аутентификация, ролевая модель) удобнее реализовывать на сервере, а не в клиенте

2. **Разделение ответственности**: 
   - Клиент отвечает только за отображение и ввод данных
   - Сервер обрабатывает бизнес-правила (например, «ребёнок не может быть без родителя» из предметной области)
   - БД гарантирует целостность данных (транзакции при бронировании, финансовые операции)

3. **Гибкость на будущее**: 
   - Если позже появится мобильное приложение для менеджеров в аэропорту, оно будет использовать тот же REST API
   - Интеграция с внешними системами (`NFR-COM-03`) проще через централизованный бэкенд

**Почему не монолит (классический)**:

- Монолит с серверным рендерингом (JSP, PHP-шаблоны) усложнит реализацию `NFR-USE-02` (адаптивный дизайн) и `NFR-USE-04` (экспорт в разные форматы)
- При добавлении мобильного клиента пришлось бы дублировать логику или делать отдельный API — лишняя работа

**Почему не микросервисы**:

- Команда разработки — одна (учебный проект), нагрузка небольшая (`NFR-PER-04`: до 10 одновременных пользователей)
- Микросервисы добавят сложность: оркестрация, распределённые транзакции, мониторинг
- Для предметной области связи между сущностями плотные (турист → группа → рейс → бронь → виза) — в микросервисах это выльется в частые межсервисные вызовы и потенциальные проблемы с согласованностью данных

**Минусы выбранного подхода и как с ними жить**:

| Минус | Как компенсируем |
|-------|-----------------|
| Единая точка отказа (сервер приложений) | Резервное копирование (`NFR-REL-02`) + быстрый деплой на новый сервер при сбое (`NFR-REL-03`) |
| Потенциальное «раздувание» бэкенда | Чёткая модульная структура кода (Раздел 5) + регулярный рефакторинг |
| Ограничения вертикального масштабирования | Архитектура позволяет позже вынести «тяжёлые» модули (отчёты, экспорт) в отдельные воркеры |

**Условие перехода на микросервисы в будущем**:
Если представительство откроет филиалы в других странах, и нагрузка вырастет в 10+ раз, или появится требование к независимому масштабированию модулей (например, грузовой учёт работает в пике сезона, а визы — равномерно), тогда имеет смысл выделить модули `Cargo`, `Reporting`, `Visa` в отдельные сервисы.

## Раздел 2 — Компоненты системы

### Детальная схема

```
┌─────────────────────────────────────────┐
│            Клиентская часть             │
│  ┌─────────────────────────────────┐   │
│  │ Веб-интерфейс (SPA)             │   │
│  │ - Forms: туристы, бронь, груз   │   │
│  │ - Tables: списки, отчёты        │   │
│  │ - Export: PDF/XLSX по кнопке    │   │
│  └─────────────────────────────────┘   │
└──────────────┬──────────────────────────┘
               │ HTTPS / JSON
               ▼
┌─────────────────────────────────────────┐
│           Сервер приложений              │
│  ┌─────────────────────────────────┐   │
│  │ REST API (бэкенд)               │   │
│  │ - Auth middleware               │   │
│  │ - Controllers (маршруты)        │   │
│  │ - Services (бизнес-логика)      │   │
│  │ - Repositories (доступ к БД)    │   │
│  └─────────────────────────────────┘   │
│                                         │
│  ┌─────────────────────────────────┐   │
│  │ Фоновые задачи (опционально)    │   │
│  │ - Генерация тяжёлых отчётов     │   │
│  │ - Экспорт больших списков       │   │
│  └─────────────────────────────────┘   │
└──────────────┬──────────────────────────┘
               │ SQL
               ▼
┌─────────────────────────────────────────┐
│           База данных                   │
│  ┌─────────────────────────────────┐   │
│  │ MySQL 8.0                        │   │
│  │ - Таблицы из Лаб 7 (14 сущностей)│  │
│  │ - Индексы для поиска (паспорт,  │   │
│  │   даты, рейсы)                   │   │
│  │ - Транзакции для финансовых     │   │
│  │   операций                       │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

### Описание компонентов

| Компонент | Что делает | Почему нужен именно в нашей системе |
|-----------|-----------|-----------------------------------|
| **Веб-интерфейс (клиент)** | Отображает формы ввода, таблицы с данными, кнопки экспорта. Валидирует ввод на фронтенде (формат паспорта, даты). | Требуется `NFR-USE-01` (интуитивный интерфейс) и `NFR-USE-02` (адаптивность). Сотрудники работают с разных устройств — в офисе, в аэропорту, в гостинице. |
| **REST API (бэкенд)** | Принимает HTTP-запросы, проверяет права доступа (`NFR-SEC-02`), выполняет бизнес-логику (например, проверка: «если турист — ребёнок, то обязательно указать родителя»), возвращает JSON. | Центральная точка применения правил предметной области. Позволяет в будущем подключить мобильный клиент без переписывания логики. |
| **База данных (MySQL)** | Хранит все сущности: туристов, группы, рейсы, брони, визы, груз, финансы. Обеспечивает целостность через FK, UNIQUE, транзакции. | Данные реляционные и сильно связаны (см. Лаб 7). Финансовые операции (`FR-FIN-01`) требуют транзакционности — `COMMIT/ROLLBACK`. |
| **Фоновые задачи (опционально)** | Асинхронно генерирует тяжёлые отчёты (например, `Запрос 13` — рентабельность за год) и экспортирует большие списки в PDF. | Чтобы не «подвешивать» интерфейс при формировании отчёта за 5 лет (требование `NFR-PER-03`: отчёт по группе — до 10 сек, но сложные аналитические запросы могут идти дольше). |

### Почему не добавили лишнего

- **Нет Redis**: Поиск туриста по паспорту (`FR-TUR-04`, `NFR-PER-01`: до 2 сек) на 50 000 записей с индексом в MySQL выполняется за миллисекунды. Кеш добавим, если мониторинг покажет узкое место.
- **Нет брокера сообщений**: Все процессы синхронные и короткие (оформление визы, бронь номера). Асинхронность нужна только для тяжёлых отчётов — это решается простым воркером, а не полноценной очередью.
- **Нет CDN**: Система внутренняя, пользователи — сотрудники представительства в одной локации. Статика (JS, CSS) кэшируется браузером.

## Раздел 3 — Оценка нагрузки (back-of-the-envelope)

### Исходные допущения

- Представительство работает в часовом поясе +3 к Москве, рабочий день 08:00–20:00 (12 часов)
- Сезонность: пик — лето и праздники, спад — межсезонье
- Команда: 5–10 сотрудников (менеджеры, бухгалтер, логист)

### Шаг 1: DAU (Daily Active Users)

```
Средний день (не пик): 5 сотрудников × 80% активности = 4 пользователя
Пиковый день (приём группы): 10 сотрудников × 100% = 10 пользователей
```

### Шаг 2: Действия пользователя в день

| Действие | Кол-во в день (среднее) | Описание |
|----------|------------------------|----------|
| Поиск туриста | 20 | По паспорту/ФИО при встрече в аэропорту |
| Создание/редактирование туриста | 10 | Внесение данных из заявки РФ |
| Бронирование номера | 5 | На группу из 30–40 человек |
| Запись на экскурсию | 15 | Массовая запись группы |
| Внесение груза | 8 | Для «шоп-туристов» |
| Финансовая операция | 10 | Внесение расходов/доходов |
| Генерация отчёта | 3 | Списки для таможни, расселения, финансовый отчёт |
| **Итого запросов к API** | **~70** | На одного активного пользователя |

### Шаг 3: RPS (Requests Per Second)

```
Средняя нагрузка:
  4 пользователя × 70 запросов = 280 запросов в день
  280 / (12 часов × 3600 сек) ≈ 0.007 RPS (среднее)

Пиковая нагрузка (×3 от среднего, плюс одновременная работа):
  10 пользователей × 70 запросов = 700 запросов в день
  Пик в течение 1 часа: 700 / 3600 ≈ 0.2 RPS

Но учтём «взрывные» запросы:
  - Генерация списка для таможни на 40 человек: 1 запрос, но тяжёлый
  - Экспорт финансового отчёта: 1 запрос, но с джойнами по 10 таблицам

Итого: пиковый RPS ≈ 2–3 (с запасом ×10 от расчётного)
```

### Шаг 4: Объём данных

```
В день (пиковый):
  - Новые туристы: 40 (одна группа)
  - Бронирования: 20 номеров
  - Записи на экскурсии: 30
  - Грузовые ведомости: 15
  - Финансовые операции: 20
  → ~125 новых записей в день

В год (200 рабочих дней):
  125 × 200 = 25 000 новых записей
  + исторические данные за прошлые годы
  → Итого: ~50 000–70 000 записей в основных таблицах

Объём в байтах:
  Средняя запись: ~1 КБ (текст + числа)
  70 000 × 1 КБ = 70 МБ «чистых» данных
  + индексы, служебная информация
  → ~200–300 МБ на БД

Вывод: база данных легко поместится на одном диске, рост за 5 лет — до 1–2 ГБ
```

### Шаг 5: Вывод по масштабированию

```
Один сервер (4 vCPU, 8 GB RAM, SSD 100 GB) справится:
  - 2–3 RPS — ничтожная нагрузка для современного бэкенда
  - 300 МБ БД — меньше 1% от доступного места
  - 10 одновременных пользователей — соответствует `NFR-PER-04`

Горизонтальное масштабирование НЕ нужно прямо сейчас.

План на будущее:
  - Если DAU вырастет до 50+ (открытие филиалов) — добавить реплику БД для чтения
  - Если отчёты начнут генерироваться дольше 10 сек (`NFR-PER-03`) — вынести в фоновые задачи
  - Если появится мобильный клиент с оффлайн-режимом — добавить кеш на клиенте
```

## Раздел 4 — Выбор стека технологий

### Сводная таблица

| Слой | Выбор | Альтернатива | Почему выбрали именно это |
|------|-------|--------------|--------------------------|
| **Бэкенд** | Python + FastAPI | Node.js + Express / Django | FastAPI: автоматическая валидация данных (через Pydantic) — экономит код для `NFR-USE-03` (валидация форм). Асинхронность «из коробки» — полезно для параллельных запросов к БД при формировании отчётов (`Запрос 15`). Django слишком «тяжёлый» для учебного проекта. |
| **База данных** | MySQL 8.0 | PostgreSQL / SQLite | MySQL: уже спроектирована схема в Лаб 7 (обратная разработка в Workbench). Поддержка `CHECK`-ограничений, транзакций, репликации. PostgreSQL мощнее, но для нашей нагрузки избыточен. SQLite не подходит для многопользовательской работы (`NFR-PER-04`). |
| **Фронтенд** | Vue 3 + Vite | React / Svelte | Vue: плавный порог входа для студентов, отличная документация на русском. Composition API удобен для переиспользуемых форм (турист, груз, бронь). Vite — быстрая сборка, что важно при частых изменениях в учебном проекте. |
| **Кеш** | Не используем (пока) | Redis | Обоснование в Разделе 2: индексы MySQL + малый объём данных делают кеш избыточным. Добавим при росте нагрузки. |
| **Фоновые задачи** | Celery + Redis (опционально) | RabbitMQ / встроенный asyncio | Celery: простой запуск воркеров для тяжёлых отчётов. Redis как брокер — минимальная настройка. RabbitMQ избыточен для 1–2 типов фоновых задач. |
| **Аутентификация** | JWT + HTTP-only cookies | OAuth2 / сессии на сервере | JWT: stateless, легко масштабируется. Cookies с флагом `HttpOnly` защищают от XSS (`NFR-SEC-03`). OAuth2 избыточен для внутренней системы. |

### Детальное обоснование ключевых выборов

**Python + FastAPI**:
- В предметной области много валидации: формат паспорта, даты, веса груза (`NFR-USE-03`). Pydantic (встроен в FastAPI) позволяет описать правила один раз и использовать их и на входе, и в ответах API.
- Асинхронные эндпоинты пригодятся, когда нужно параллельно запросить данные из нескольких таблиц для отчёта (`Запрос 15`: туристы + гостиницы + груз + бирки).
- Альтернатива Django: мощный, но «монолитный» фреймворк. Для учебного проекта с чёткими модулями (`FR-TUR`, `FR-VIS` и т.д.) легковесный FastAPI даёт больше гибкости.

**MySQL 8.0**:
- Все связи между сущностями (Лаб 7) — реляционные, с внешними ключами. MySQL отлично справляется с такими данными.
- Транзакции критичны для финансовых операций (`FR-FIN-01`): если при внесении расхода произошла ошибка, вся операция должна откатиться.
- Обратная разработка из DDL-скрипта в MySQL Workbench — стандартная практика для учебных проектов.

**Vue 3**:
- Компонентная архитектура: можно создать переиспользуемые компоненты `TouristForm`, `CargoTable`, `ReportExport`, что ускоряет разработку.
- Реактивность «из коробки»: при изменении категории туриста (`leisure`/`cargo`/`child`) автоматически скрываются/показываются нужные поля — реализация `FR-TUR-02`.
- Альтернатива React: мощнее, но требует больше настройки (Redux, роутинг). Для проекта с 5–10 экранами Vue оптимален.

    ├── visa.py
    └── ... (по одной на сущность)

### Как слои взаимодействуют (пример: запись туриста на экскурсию)

```
1. Клиент → POST /api/v1/excursions/bookings
   { "tourist_id": 123, "excursion_id": 456 }

2. Controller (endpoints/excursions.py):
   - Проверяет токен (deps.get_current_user)
   - Валидирует входные данные через Pydantic-схему
   - Вызывает: ExcursionService.create_booking(...)

3. Service (services/excursion_service.py):
   - Проверяет бизнес-правила:
     * Есть ли свободные места в экскурсии? (Запрос к Repository)
     * Не превышен ли возраст туриста для min_age? (FR-EXC-01)
     * Если турист — ребёнок, подтверждён ли родитель? (предметная область)
   - Если всё ок → вызывает Repository.create()

4. Repository (repositories/excursion_booking_repo.py):
   - Формирует SQL-запрос через SQLAlchemy
   - Выполняет в транзакции (на случай, если позже добавим аудит)

5. Ответ:
   - Сервис возвращает созданную запись
   - Контроллер сериализует её в JSON
   - Клиент получает 201 Created + данные
```

### Модули бэкенда (соответствие Лаб 5)

Каждый модуль в `api/v1/endpoints/` и `services/` соответствует функциональному модулю из Лаб 5:

| Модуль кода | Соответствие ФТ | Что внутри |
|-------------|----------------|-----------|
| `tourists.*` | FR-TUR-01..04 | Регистрация, категоризация, поиск, история |
| `visas.*` | FR-VIS-01..03 | Генерация пакетов, статусы, таможенные списки |
| `hotels.*` | FR-HTL-01..03 | Бронь, списки расселения, загрузка отелей |
| `excursions.*` | FR-EXC-01..04 | Каталог, запись, списки для агентств, аналитика |
| `cargo.*` | FR-CRG-01..03 | Весовые ведомости, отгрузочные документы, статистика |
| `finances.*` | FR-FIN-01..04 | Учёт операций, отчёты, рентабельность |
| `reports.*` | FR-ANL + Запросы 1–15 | Агрегация данных для экспорта в PDF/Excel |

### Почему именно такая структура

- **Разделение ответственности**: Контроллер не лезет в БД напрямую, сервис не знает про HTTP. Это упрощает тестирование и рефакторинг.
- **Модульность по предметной области**: Если завтра изменится правило оформления виз (`FR-VIS-02`), правим только `visa_service.py`, не затрагивая модуль груза.
- **Соответствие `NFR-MNT-01` (модульная архитектура)**: Каждый функциональный блок изолирован, что упрощает добавление новых функций (например, интеграция с внешним порталом миграционной службы).

## Итог

Выбранная архитектура (клиент–сервер, 3 уровня) оптимальна для учебного проекта:
- Покрывает все функциональные (`Лаб 5`) и нефункциональные (`Лаб 6`) требования
- Масштабируется вертикально при росте нагрузки
- Позволяет легко добавить мобильный клиент или внешнюю интеграцию в будущем
- Структура кода соответствует модульности предметной области, что упрощает поддержку
